# 테스트 마스터 플랜

`when-should-we-meet` 프로젝트의 장기 테스트 전략을 정의,
어떤 테스트 층위를 운영하는지, 각 층위가 무엇을 증명하는지, 기능 변경 시 어떤 문서를 함께 갱신해야 하는지를 이 문서를 기준으로 판단한다.

<strong>버전 : </strong> v1

<strong>생성 날짜 : </strong> 2026-05-19

<strong>최신 업데이트 날짜 : </strong> 2026-05-19

<br/>

## 목표

- 핵심 플로우를 랜딩부터 공유까지 안정적으로 보호
- 결정적인 제품 동작은 가능한 한 자동화 테스트로 소유한다.
- 외부 SDK의 실제 화면 전환이나 디바이스 의존 동작은 수동 스모크 범위로 남긴다.
- 테스트를 일회성 산출물이 아니라 제품 계약으로 취급한다.

<br/>

## 보호 대상 플로우

- 랜딩 진입과 방 생성
- 초대코드 참여와 딥링크 진입
- 닉네임 참여와 참가자 복원
- 달력 선택, 불가능 모드, 요일 규칙, 랭킹 업데이트
- 호스트 전용 액션: 참가자 제거, 방 삭제
- 공유 플로우: 입장 코드 복사, 방 공유, 랭킹 공유, Kakao 어댑터 호출
- Firestore 기반 실시간 동기화와 새로고침 후 복원

<br/>

## 테스트 층위

### 1. 유닛 테스트

<strong>목적</strong>

- 순수 유틸, 라우트 파싱, 초대코드 정규화, 날짜 계산, 랭킹 계산,
  선택 모드 변환을 검증한다.

<strong>원칙</strong>

- 입력/출력으로 명확하게 표현 가능한 로직은 우선 유닛 테스트로 검증한다.
- 빠르고 환경 의존이 없어야 한다.

<strong>주요 명령어</strong>

- `npm run test:unit`

### 2. 로컬 E2E

<strong>목적</strong>

- Firebase 없이 단일 클라이언트 기준 핵심 UI 플로우를 검증한다.
- 로컬 폴백, 랜딩 UX, 방 생성, 참여 유효성 검사, 클립보드, Web Share API
  동작을 커버한다.

<strong>원칙</strong>

- UI 플로우와 접근성 셀렉터 검증용으로 사용한다.
- Firestore 영속성이나 실시간 동기화를 이 층위로 주장하지 않는다.

<strong>주요 명령어</strong>

- `npm run test:e2e:local`

### 3. 카카오 계약 E2E

<strong>목적</strong>

- 앱이 Kakao 공유 분기로 진입하는지와, Kakao SDK 어댑터를 올바른
  payload로 호출하는지를 검증한다.

<strong>원칙</strong>

- `window.Kakao`를 mock 하고 `init`과 `Share.sendDefault(...)`를 검증한다.
- 실제 Kakao UI 또는 앱 전환은 CI 소유 범위로 보지 않는다.

<strong>주요 명령어</strong>

- `npm run test:e2e:kakao`

### 4.Firebase Emulator E2E

<strong>목적</strong>

- Firestore 기반 방 플로우, 참가자 생명주기, 실시간 동기화, 새로고침 후
  복원, 문서 정리, 리스너 복원력을 검증한다.

<strong>원칙</strong>

- 멀티 클라이언트와 영속성 검증은 emulator를 기본 자동화 타깃으로 둔다.
- 먼저 사용자 가시 결과를 검증하고, 필요할 때만 문서 단언을 추가한다.

<strong>주요 명령어</strong>

- `npm run test:e2e:firebase`

### 5.수동 스모크

<strong>목적</strong>

- 릴리즈 빌드가 실제 브라우저/디바이스 기준으로 살아 있는지 짧게 확인한다.

<strong>범위</strong>

- 실제 Kakao 공유 UI 또는 앱 전환
- 실제 모바일 디바이스 레이아웃 점검
- 배포 도메인 기준 공유 URL 확인
- 자동화 가치가 낮은 시각적 회귀 확인

<strong>참고 문서</strong>

- `docs/manual-qa-checklist.md`

## 환경 전략

### 로컬 폴백 환경

사용 시점:

- 백엔드 상태 없이 UI와 상호작용만 빠르게 검증할 때

비고:

- Firebase 환경변수는 의도적으로 비워둔다.
- 이 모드의 테스트는 Firestore 커버리지를 주장하면 안 된다.

### Firebase Emulator 환경

사용 시점:

- 실시간 동기화, 영속성, 참가자 문서, 삭제 후 정리를 검증할 때

비고:

- Firestore emulator 실행에는 JDK 21이 필요하다.
- 백엔드 연동 E2E의 기본 자동화 타깃이다.

### 실제 브라우저/디바이스 환경

사용 시점:

- 외부 UI 전환이나 배포 도메인 동작을 확인할 때

비고:

- 이 영역은 수동 스모크 범위다.

## 역할과 책임

### 기능 작업자

- 기능 변경과 같은 PR에서 테스트를 추가하거나 수정한다.
- 유저 플로우나 제품 계약이 바뀌면 변경별 테스트 플랜도 같이 갱신한다.
- 자동화 갭이나 수동 전용 항목이 있으면 PR 본문에 남긴다.

### 리뷰어 또는 QA 담당

- 변경된 플로우가 올바른 테스트 층위에 올라갔는지 확인한다.
- 구현 세부사항만 따라가는 깨지기 쉬운 테스트를 지양한다.
- 이전/신규 플로우가 다르면 레거시 처리 방식이 문서에 남았는지 확인한다.

## 이 마스터 플랜을 수정해야 하는 경우

다음처럼 장기 전략이 바뀔 때만 이 문서를 수정한다.

- 새로운 테스트 층위가 추가될 때
- CI 소유 범위가 바뀔 때
- 외부 연동이 자동화 범위와 수동 범위 사이를 이동할 때
- merge blocking 기준이 바뀔 때

일반적인 기능 단위 플로우 변경은 여기서 다루지 않고, 변경별 테스트 플랜에
기록한다.

## 변경별 테스트 플랜 사용 기준

아래 경우 `skills/qa/test-plan-template.md`를 사용한다.

- 유저 플로우가 바뀔 때
- 레거시 플로우를 제거하거나 일정 기간 유지할 때
- 새로운 리스크 영역이 생길 때
- 자동화가 추가, 삭제, 이동될 때

변경별 테스트 플랜에는 다음을 기록한다.

- 이전 플로우
- 신규 플로우
- 레거시 처리
- 자동화 범위
- 수동 스모크 범위
- 검증 명령어
- 배포 후 모니터링 포인트

## 현재 기준 테스트 관련 주요 명령어

- `npm test`
- `npm run test:e2e:all`
- `npm run test:e2e:local`
- `npm run test:e2e:kakao`
- `npm run test:e2e:firebase`
- `npm run test:unit`
- `npm run lint`
- `npm run build`
