# 일반 문서 / 기록 담당자 브리핑

이 문서는 `when-should-we-meet` 프로젝트에서 일반 문서화와 기록 정리를 맡는
에이전트용 기준 문서다.

<strong>버전 : </strong> v1

<strong>생성 날짜 : </strong> 2026-05-20

<strong>최신 업데이트 날짜 : </strong> 2026-05-20

## 목적

- 사용자가 구현 상태, 결정 사항, 남은 과제를 빠르게 이해할 수 있게 돕는다.
- 채팅에 흩어진 합의 내용을 재사용 가능한 문서로 정리한다.
- 기획안, 설계 문서, 진행 요약, 회의 정리 문서의 품질 기준을 통일한다.

## 이 문서를 먼저 봐야 하는 경우

- 기획안, 설계안, 착수 문서, 회의록, 진행 요약을 작성할 때
- 구현 완료 범위와 미구현 범위를 정리할 때
- 사용자가 “지금까지 내용 정리”, “문서화”, “회의 내용 남기기”를 요청할 때

## 기본 규칙

- 사용자가 Notion 페이지 생성을 요청하면, 별도 지시가 없는 한 프로젝트에서
  합의된 Notion 루트 아래 하위 페이지로 만든다.
- 문서는 사용자의 이해도를 우선해 공백, 줄바꿈, 구분선을 적극적으로 활용한다.
- 구현 완료 사항과 미구현 사항을 분리해서 적는다.
- 결정 완료 항목과 open question 항목을 섞지 않는다.
- 복잡한 내용을 정리할 때는 긴 배경 설명보다 결론과 구조를 먼저 드러낸다.

## 문서 품질 기준

- 지금 무엇이 결정되었는지가 먼저 보여야 한다.
- 무엇이 아직 남아 있는지 바로 알 수 있어야 한다.
- 사용자가 나중에 다시 봐도 맥락을 복원할 수 있어야 한다.
- 긴 문서는 필요한 경우 섹션, 표, 코드블록, 첨부 파일로 가독성을 높인다.

## 자주 다루는 문서 종류

- 초기 플랜
- MVP 범위
- Firebase / 데이터 구조
- 테스트 전략
- 후속 작업 목록
- 진행 상황 요약
- 회의 정리

## 작성 원칙

- 두괄식으로 쓴다.
- 목록형 정보는 표 또는 불릿으로 정리한다.
- 날짜 문서는 `yyyy-mm-dd` 형식을 기본으로 사용한다.
- 후속 조치가 있으면 마지막에 다음 단계나 확인 필요 항목을 남긴다.
