# 엔지니어링 워크플로우

이 문서는 `when-should-we-meet` 프로젝트의 기본 작업 규칙을 정의한다.
<strong>버전 : </strong> v2

<strong>생성 날짜 : </strong> 2026-04-15

<strong>최신 업데이트 날짜 : </strong> 2026-05-19

## 공통 작업 규칙

- PR을 열면 PR 리뷰어 에이전트가 해당 GitHub PR 코멘트에 리뷰 기록을 남긴다.

## 구현 규칙

- 로컬 상태와 Firestore 상태의 책임 경계를 분명하게 유지한다.
- 모바일 전용 레이아웃을 깨는 desktop/tablet 중심 변경을 피한다.
- Firestore rules는 접근 정책의 일부이므로 Git에 포함해 관리한다.
- secrets, 실제 민감 정보, 운영 데이터 dump는 커밋하지 않는다.
- 개발 착수전에 구현내용에 대한 보고 후에 개발에 착수한다. e.g.) ~~ 관련 내용 ~ 스택 or ~ 방법으로 착수합니다.

## PR 생성 이후

- 현재 열린 PR의 머지와 무관한 작업이라면 바로 최신 기준 브랜치에서 새 브랜치를 만들어 이어서 진행한다.

## 검증 규칙

- 사용자 플로우, 상태 모델, 외부 연동이 바뀌면 관련 테스트나 QA 문서도 함께 갱신한다.
- flaky한 자동화보다 결정적인 assertion과 수동 smoke 범위 구분을 우선한다.

## 우선 판단 기준

충돌하는 요구가 있으면 아래 순서로 판단한다.

1. 사용자 지시
2. 현재 작업의 제품 리스크
3. 모바일 UX 일관성
4. 상태 모델 안정성
5. 구현 편의성
